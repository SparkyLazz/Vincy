# Console — Build Workflow (Fidelity-First, No-Rush)

> How this project gets built so the shipped app matches the design 100%, and how each phase gets verified before the next one starts. This governs every Claude Code session working on Console.

---

## Prime directive

**Extraction, not recreation.** The prototypes and design library are not references to redraw from memory — they are the literal source of the app's CSS and, where possible, its markup. A screen is "done" when it reuses the exact tokens, exact component classes, and exact structure already proven in the prototype, wired up to real data and behavior. If a build ever produces a screen that *looks* right but was rewritten from a description of the screen, that's not 100% fidelity — it's a good guess.

**No rushing.** One phase at a time, fully verified, before starting the next. A phase that "mostly works" is not done. Shipping five modules that are each 80% right is worse than shipping two modules that are 100% right — the gaps compound and get harder to find later.

---

## What we learned extracting the Phase 1 bundle (read this before touching CSS)

While pulling `tokens.css` / `base.css` / `components.css` out of the prototypes, real drift showed up between files that were assumed to be identical:

- **Design tokens** (`:root` / `[data-theme="dark"]` color and font variables) are byte-identical in value across the design library and all 5 prototypes — only whitespace formatting differs. Safe to treat as a single canonical source (`tokens.css`, pulled from the design library).
- **Shell chrome** (sidebar, topbar, cmd-trigger, theme-toggle) is identical across Tasks/Schedule/Habits/Finance (verified by checksum) and semantically identical in Today (older formatting, same values). Today was used as the `base.css` source since Phase 1 = Foundation + Today.
- **Real conflicts found and resolved:** `.segmented`, `.breadcrumb`, `.card`, `.pill`, and `.priority` are each defined *twice* with different values — once in the design library's component showcase, once inside a shipped prototype. Where a class is part of the topbar/shell (`.segmented`, `.breadcrumb`), the shell's version wins — it's what's actually on screen in every prototype. Where a class is a generic content component (`.card`, `.pill`, `.priority`), the design library's version wins, since the library is explicitly the "final" spec and the small prototype deviations (11px vs 12px priority bars, etc.) look like unpatched legacy drift from Today being built first.
- Every one of these decisions is logged as a comment at the point of conflict in `components.css` / `base.css`, not just in this file — so the reasoning survives even if someone reads the CSS in isolation.

**The lesson this sets for every future phase:** don't assume two files that "should" be the same actually are. Diff or checksum before extracting. When they disagree, decide deliberately and write down why, in the file itself.

## Phase 2 browser verification (2026-07-04)

User ran the manual `file://` pass this session (capture bar, all 8 views, detail pane, subtasks, shortcuts, both themes) — reported all correct except one visual defect.

- **Real bug found and fixed — stray native scrollbar.** A small unstyled up/down-arrow scrollbar (no visible track) appeared pinned near the page-head-row's action buttons. This is exactly the failure mode `base.css`'s own RESET comment already predicted (`html, body { overflow: hidden }` alone isn't reliably propagated to the viewport in every Chromium build/zoom level) — the first pass's fix only covered `html::-webkit-scrollbar`, not `body`, and had no Firefox fallback. Closed all three gaps: `overflow: clip` added alongside `overflow: hidden` on both `html` and `body` (unlike `hidden`, `clip` never establishes a scroll container at all, so there's nothing left for the quirk to draw a scrollbar for), `::-webkit-scrollbar { display: none }` added to `body` (previously only `html`), and `scrollbar-width: none` added to both for Firefox (which has no WebKit pseudo-element).

## Per-phase loop

1. **Spec** — write `docs/phase_prompts/phase_N_<name>.md`: scope, data model touched, views + behavior, explicit acceptance criteria, explicit exclusions (what NOT to build yet).
2. **Extract** — before writing any new markup, check whether the prototype for this phase already defines the classes/structure needed. Reuse its DOM shape; don't invent a parallel structure that happens to look similar.
3. **Diff new shared patterns against what's already in the bundle** — if this phase's prototype introduces a header/list/card pattern not yet in `base.css`/`components.css` (expected for Phase 2+, see the Today-only limitation above), extract it the same way Phase 1 did: checksum across the files that use it, resolve conflicts explicitly, add to the shared CSS with a comment, and note it in this file's "what we learned" log.
4. **Wire** — bind IndexedDB, routing, interactivity. Markup and CSS classes stay untouched unless something is factually broken (e.g. a missing `@keyframes` — see the drift log below).
5. **Verify** — run the `visual_contract.md` checklist against the built screen. Every item must pass, not "mostly" pass.
6. **Log drift** — anything that had to diverge from a prototype (a bug in the prototype, a value that couldn't be reused as-is) gets written down with the reason, in the relevant CSS file and in this doc if it's structural. Silent divergence is the failure mode this whole workflow exists to prevent.
7. **Gate** — only move to the next phase once 5 and 6 are done. Don't start Phase N+1 with known gaps in Phase N "to fix later."

---

## What "100% fidelity" means in practice

Same CSS custom properties, same component classes, same DOM structure as the prototype for that screen. The only differences should be: real data instead of mock data, and working interactivity instead of static markup. Restyling, "improving," or reinterpreting the design mid-build is out of scope for an implementation phase — design changes go back through a new prototype, not through ad hoc changes while wiring up a module.

---

## Anti-patterns (things that produced drift once already — don't repeat them)

- Inlining a one-off style instead of extending `components.css` — this is exactly how `.card`/`.pill`/`.priority` ended up with two silently conflicting definitions in the first place.
- Guessing a color, spacing, or type value instead of reading it from `tokens.css`.
- Building a screen from the written spec/description instead of the actual prototype `.html` file.
- Marking a phase done with a known visual mismatch, intending to "fix it in polish."
- Assuming two prototype files share a component just because they use the same class name — verify with a diff or checksum first, the same way `.segmented` and `.breadcrumb` turned out not to match.
