## Resolved conflicts — read before writing any layout/component code

Finance's prototype is structurally different from Tasks/Schedule/Habits in ways that matter before writing a line of markup:

1. **Not a "flush" module — reuses Today's plain `.content`/`.content-inner` layout.** Already checked and logged in `base.css`'s own header comment (from the Phase 2 checksum pass, which grepped all four prototypes ahead of time): Finance's prototype doesn't use the `page-head-row`/`view-tabs-row`/`metrics-row`/`twopane-row` shell pattern Tasks/Schedule/Habits share at all — it uses `.page-head`/`.view-tabs`/`.metrics-strip`/`.two-col`/`.card`, the same family of classes Today dashboard's own Phase 1 components already extracted (`.page-head`, `.card`/`.card-head`/`.card-title`/`.card-sub`, `.two-col`/`.two-col.flip`, all already in `components.css`). **Decision: `Console.modules.finance` does NOT set `layout: 'flush'`** — it renders inside `.content`/`.content-inner` like Today does, reusing those existing classes, not Tasks/Schedule/Habits' twopane shell. Finance's own `.content` padding (22px top) is 4px off the canonical Today-extracted value (26px) — same unpatched-drift call as Phase 1's priority-bar diff: canonical wins, prototype's value not reproduced.
2. **`.card`/`.card-head`/`.card-body` name collision, resolved as `.pane`, not a new shape.** Finance's prototype defines its own `.card { border-radius:13px; overflow:hidden; display:flex; flex-direction:column; }` / `.card-head { padding:14px 18px 12px; border-bottom:1px solid var(--line-1); }` / `.card-body { padding:12px 18px 16px; }` — this is a *different structural shape* from the canonical `.card` already in `components.css` (single padded block, no head/body split, no border-bottom, no flex-column). But it's nearly byte-identical to the existing `.pane`/`.pane-head`/`.pane-body` (13px radius, overflow hidden, flex column, bordered head) already extracted for Tasks/Schedule/Habits' twopane shells. **Decision: Finance's Envelopes card, Recent-transactions card, and every other card-shaped section reuse `.pane`/`.pane-head`/`.pane-title`/`.pane-meta`/`.pane-body`** (with a scoped `padding` added to whatever wraps the actual content, same pattern as Phase 4's `.pane-body > .habits-grid`) — not a second, conflicting `.card` definition. This is the same class of "same name, different shape" drift Phase 1 resolved for `.segmented`/`.breadcrumb`, just inverted (different name, same shape).
3. **`.metrics-strip` reuses `.metric`'s inner classes, scoped.** Finance's metric cells (`.mnum`/`.mlbl`/`.mtrend`) are the same concept as the canonical `.metric` (already in `base.css`, used by Tasks/Schedule/Habits' `.metrics-row`) but with different padding/radius/font-size. Reuse the same inner class names, scoped under a new `.metrics-strip` wrapper for the value deltas — same precedent as `.pane .task-row`/`.pane .list-row.event` scoping, not a full parallel component. `.mnum.pos`/`.neg`/`.warn` (color modifiers) are added at the top level (generic, like Phase 4's `.mnum.teal`/`.amber`), not scoped to Finance only.
4. **`.txn-row` (Overview's compact widget) vs. `.list-row.txn` (unused since Phase 1, first real consumer here).** The already-defined `.list-row.txn { grid-template-columns: 1fr 110px 110px auto; }` (4 columns) doesn't match Overview's actual compact `.txn-row` (3 columns: date/info/amount) — same "found an unused generic template with the wrong column count for this context" situation as Phase 3's `.list-row.event` discovery. **Decision:** Overview's "Recent transactions" card keeps its prototype's exact compact `.txn-row` shape (extraction, not recreation, since a prototype exists for this exact widget); the dedicated **Transactions view** (no prototype) is built fresh using `.list-row.txn`'s existing 4-column template (date / title+meta / envelope+category / amount) — the unused template finally gets a real, appropriate consumer instead of being left to silently mismatch anything.

## Scope

Finance = multi-currency transactions against a YNAB-style envelope budget, a subscription audit, and cross-domain cost-per-focus-hour reporting.

1. **Transactions** — `transactions` store: `id`, `date`, `title`, `amount` (signed, in base/home currency — positive = income, negative = expense, matches `.txn-amount.pos/.neg`), `currency` (ISO code, e.g. `'USD'`/`'EUR'`; `null`/omitted means already in base currency), `fx_rate` (only set when `currency` isn't base — `amount = original_amount * fx_rate`), `original_amount` (only set when `currency` isn't base, for the `.fx-orig` display), `envelope_id` (optional — `null` for unbudgeted/income transactions), `project_id` (optional cross-link to `projects`), `recurring` (bool — a subscription instance, see #3), `recur_period` (`'monthly'` etc., only when `recurring`), `usage_count` (integer, only meaningful when `recurring` — see #3), `notes`. "New transaction" button + `N` shortcut open a modal (reuse Schedule's `.modal-overlay`/`.modal`/`.modal-field` system, same precedent Phase 4 set for its own habit/log forms — don't invent a new modal shell). `⌘I` opens an Import affordance (file picker → JSON, matching the existing `db.uploadImport` shape used for full-app backups is out of scope here — a Finance-only CSV/JSON import is a nice-to-have; if time-boxed, a working "New transaction" form alone satisfies the acceptance criteria and Import can render as a disabled/stub button, flagged rather than silently faked).
2. **Envelopes (YNAB-style)** — `envelopes` store: `id`, `category_id` (join `categories`), `period` (`'YYYY-MM'`, one envelope row per category per month — allocations reset monthly, matching real YNAB behavior), `allocated` (amount budgeted this period), `rollover` (bool — unused allocation carries into next period's To-Be-Budgeted pool). `categories` store gets two new (undocumented-until-now, non-indexed, same precedent as `theme_days.label` being "a label, not a type") fields: `group` (one of `'fixed'/'living'/'personal'/'savings'` — the prototype's `env-section` groupings; a label, not a strict enum enforced elsewhere, since Reports/Analytics may want to add groups later) and `goal_total` (optional, only meaningful for `group: 'savings'` categories — the lifetime savings target, e.g. "$1,200 goal" for Vacation). **`spent` and `remaining` are always computed from `transactions` at read time, never stored** — same architectural call Phase 4 made for consistency (`consistency30` computed on demand, not cached) — `spent` = sum of `abs(amount)` for transactions in that envelope for that period; for a `savings` envelope, "total saved so far" sums across *all* periods' envelopes for that category, not just the current month. **To-Be-Budgeted**: `income this period` − `sum(allocated this period across all envelopes)` + `rollover pool` (sum of `allocated − spent` for every `rollover: true` envelope from the immediately preceding period, only if positive — a simplified but real YNAB rollover rule; if this needs more nuance later, keep it simple here and log the heuristic, same discipline Schedule used for theme-day adherence). `E` shortcut on a selected envelope opens its edit modal (allocated amount, rollover toggle); `F` ("Fund envelope") is a quick modal to move an amount from To-Be-Budgeted into a chosen envelope's `allocated` for the period.
3. **Subscriptions (modeled via recurring transactions, no separate store)** — `Console_Features_List.md`'s 15-store data model has no dedicated "subscriptions" store; a subscription is a `transactions` row with `recurring: true` (its most recent period's instance) grouped by `title`/`envelope_id`. Usage tracking has no external telemetry to read from (nothing in this app can know if you actually opened Netflix) — **`usage_count` is a manual, user-logged integer** on that period's recurring transaction row, incremented via a "+1 use" button in the Subscriptions view. This is an honest, real (if manually-maintained) number, not a fabricated one — flagged here so it isn't mistaken for automatic tracking later.
4. **Cost per focus hour (cross-domain, read-only)** — `fixed spend this period` (sum of `spent` across every `group: 'fixed'` envelope) ÷ `focus hours logged this period` (sum of `focus_sessions.duration_min` for sessions in this period, `/60`) — legitimately `0`/`Infinity`-guarded to `—` until Focus (Phase 6) writes real sessions, same honest-empty-state precedent as Phase 4's Focus-session cross-domain count. Rendered as the `.insight-card.info` "Cost per focus hour" callout, with a real (if currently flat/empty) sparkline built from the last 6 periods' values, not the prototype's hardcoded path.
5. **Subscription audit (insight callout)** — the `.insight-card.warn` callout: subscriptions (`recurring: true` transactions) with `usage_count` below a simple threshold (e.g. `< 3` this period) — real computed list from real `usage_count` values, not mock rows. Simple threshold, not a detector model (`subscription_low_use` is Insights/Phase 7's own scored detector job — this phase's callout is a simpler, real-but-unscored version of the same idea, same "simple heuristic now, richer model later" split Phase 4 drew between its own Correlations tab and Analytics' future Habit↔Output view).
6. **Hourly-equivalent toggle** — a view-state toggle (not persisted — same as the topbar's existing 7d/30d/90d/all segmented control, which Phase 1 logged as visual-only until Analytics) that, when active, renders every dollar amount across Finance as an hours-of-work equivalent instead, using `net income this period ÷ focus hours logged this period` as the conversion rate (falls back to a flat, clearly-labeled "no rate yet" state when focus hours are `0`, rather than dividing by zero or fabricating a rate).
7. **5 views via `.view-tabs`**: **Overview** (prototype's shown view — metrics strip, To-Be-Budgeted banner, envelopes + recent-transactions two-col, insight callouts), **Transactions** (new, no prototype — flat chronological list using `.list-row.txn`'s 4-column template per Conflict #4 above, filterable by envelope/date, `N` still opens New Transaction), **Envelopes** (new, no prototype — the full grid of `.env-card`s grouped by `.env-section`, same components as Overview's envelope card but not truncated/summarized, plus the full To-Be-Budgeted banner and a category-group editor), **Subscriptions** (new, no prototype — flat list of recurring transactions with usage counts and a "+1 use" action per #3), **Reports** (new, no prototype — income vs. expense trend over the last 6 periods, cost-per-focus-hour trend, kept simple per the "richer trend math is Analytics' job" exclusion below).
8. **Metrics strip** (`.metrics-strip`, real data, matches prototype's 4 metrics on Overview): income this period, expenses this period (+ "% of income"), net/savings rate (+ "%"), hourly-equivalent rate (+ "Nh logged"). Real month-over-month trend deltas computed the same way Phase 4 computed real 30/60/90-day-ago deltas (evaluate the same computation against the previous period instead of storing a separate snapshot).
9. **Keyboard shortcuts** (scoped to Finance active, palette closed, same collision-avoidance discipline as Phases 2–4):
   - `N` — new transaction
   - `E` — edit the selected envelope
   - `F` — fund envelope (move To-Be-Budgeted into an envelope)
   - `J` / `K` — navigate selection (transactions/envelopes list, depending on view)
   - `Enter` — open the selected row/card
   - `⌘E` — edit the selected transaction/envelope (same key as "edit," `⌘`-qualified for the already-selected-item case, mirroring every prior phase's own `⌘E`)
   - `⌘I` — import
   - `?` stays inert this phase (Settings' job, Phase 8 — same deferral every prior phase logged)
   - Month navigation (`‹`/`this month`/`›` in `.month-nav`) is click-only this phase, no dedicated keys reserved (none of the prototype's own `kbd-hints` bind one either).
10. **Sidebar Finance count**: `app.js`'s `refreshNavCounts` currently has no Finance line — add one (pick a real number, e.g. envelopes needing attention this period — over-spent or unfunded — and note the choice, same as every prior phase's own sidebar-count decision).
11. **Today dashboard cross-check**: `js/modules/today/index.js`'s `buildMoneyCard` (built in Phase 1 against an always-empty store) reads `e.target`/`e.spent` as if they were real envelope fields — but this phase's real schema has neither (`allocated` is the field name, and `spent` is computed, not stored). Re-verify and fix per the same discipline Phase 3 applied to the calendar card and Phase 4 applied to the habits card.

## Explicitly out of scope for this phase

- **Full CSV/bank import parsing** — a working "New transaction" modal is the bar; a real file-picker → parser import can be a stubbed/disabled affordance if time-boxed, flagged rather than silently faked as working.
- **Automatic subscription-usage detection** — `usage_count` is manually logged by the user this phase; any real automatic usage signal (e.g. from Focus session tags) is a future cross-domain enhancement, not this phase's job.
- **Statistically rigorous trend/correlation math in Reports** — simple period-over-period real numbers are the bar; deeper trend modeling is Analytics' (Phase 7) job.
- **Scored subscription/overspend detectors** (`subscription_low_use`, `envelope_overspend_pattern`) — those are Insights' (Phase 7) job; this phase's callouts are simpler, real-but-unscored versions of the same ideas, per Conflict/Scope note #5.
