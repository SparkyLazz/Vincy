# Phase 7 — Analytics

No prototype exists for this phase (`Console_Features_List.md` §7 lists 11 view names and nothing else — no data model, no computation, no layout). Same situation Focus was in for Phase 6: every view here is built fresh from already-proven atomic components, following `docs/Console_Workflow.md`'s per-phase loop (spec → extract/design → wire → verify → log drift → gate). Fully verify before Phase 8 (Insights) starts.

Unlike every prior phase, Analytics introduces **no new IndexedDB store**. It is a pure read/aggregation layer over the 13 stores every other phase already writes to (`tasks`, `projects`, `events`, `event_logs`, `theme_days`, `habits`, `habit_logs`, `transactions`, `envelopes`, `categories`, `focus_sessions`). `db.js` needs zero changes.

## Design decisions

1. **No `layout: 'flush'`.** Same call Phase 5/6 made — Analytics' 11 views are cards/charts/tables, not a list+detail split. Renders inside `.content`/`.content-inner`.

2. **The topbar's 7d/30d/90d/all segmented control goes live, scoped to Analytics only.** `js/app.js`'s `wireTopbarSegmented()` has said "Real range-filtering lands with Analytics (Phase 7)" since Phase 1 — this is that wiring. But the control is global chrome, visible on every screen, and retrofitting live range-filtering into Today/Tasks/Schedule/Habits/Finance/Focus would mean touching 6 already-verified, gated modules for a feature none of them asked for or were built against (Finance has its own month-nav, Schedule its own date-nav — neither reads a global range). So: the control becomes a real, persisted (`Console.db.setPref`) global range value, but **only Analytics reads it.** Every other module ignores it exactly as it does today. This is the same "don't regress an already-verified phase for a new phase's convenience" discipline as every prior phase's conflict log.

3. **Custom range is a 5th segmented option, not a redesign of the control.** The features list says "7d / 30d / 90d / all / custom range." Added a 5th `<button>custom</button>` to the existing `.segmented` group in `index.html`; clicking it opens a small popover (two `<input type="date">`s + Apply) built from existing atomic pieces (`.modal-field`/`.input`/`.btn`), not a new date-picker component. Selecting a custom range sets the same persisted global range value as the preset buttons (`{ preset: 'custom', start: iso, end: iso }` vs `{ preset: '30d' }` etc.) so Analytics' views don't need two different code paths.

4. **No charting library — every visualization is hand-rolled from what's already proven.** This app is vanilla HTML/CSS/JS with no build step; nothing like Chart.js is loaded. Habits' Phase 4 detail panel already solved this exact problem for its consistency trend (`consistencyChartSvg` — inline `<svg viewBox="0 0 400 60">` with a `<polyline>` + gradient-filled `<path>` area, computed in JS from real data points, no library). Every trend-line view in this phase (#6, #9, #10, #11 below) reuses that exact technique, generalized to N points instead of 4 fixed ones. Scatter views (#1, #4) use absolute-positioned dots inside a bordered box (`left`/`bottom` computed inline per dot — legitimate per the project's own precedent that computed *data-driven* values, unlike static design values, are allowed inline: Finance's `.ebar .fill { width: X% }`, Habits' `.hbar .fill { width: X% }`). Heatmap (#3) is a CSS grid of cells with a computed inline `opacity`. Paired/grouped bars (#2, #5, #8) get one genuinely new component, `.dual-bar` (two `.fill`-style segments stacked or side-by-side) — no existing component does this.

5. **No new "scored model."** This project has already refused scored/predictive models three times (Habits' consistency = real rolling %, not a streak-multiplier score; Finance's subscription audit = a real usage-count threshold, not a scored risk model; Schedule's theme-day adherence = a one-line real heuristic). Analytics' vaguest view (#11, Energy Index) gets the same treatment: a transparent, human-checkable average of real percentages, not a weighted/tuned formula.

6. **Every computation runs over the *current* range (from decision #2/#3) directly against raw store rows** — not through Finance's or Schedule's own module-private, calendar-month-bucketed helper functions (`costPerFocusHour(period)`, `hourlyRate(period)`, `adherencePct`, etc.), since those operate on a `"YYYY-MM"` period string, not an arbitrary day range. Where Analytics needs the *same formula* Finance already uses (cost-per-focus-hour, hourly-equivalent), the formula is duplicated against a generalized date-range instead of extracted — same "don't touch an already-verified module's private internals for a new consumer" call Phase 6 made for Schedule's `TYPES`.

## The 11 views, precisely defined

Each of these had zero definition beyond a name before this phase. All operate over the active range from decision #2/#3 unless stated otherwise.

1. **Project Investment Matrix** — one dot per non-archived project. X = money invested (sum of `abs(transaction.amount)` where `amount < 0` and `transaction.project_id` matches, in range). Y = time invested in hours (event durations with matching `project_id`, plus `focus_sessions.duration_min` linked via `task_id → task.project_id` or `event_id → event.project_id`). Dot size = that project's open task count. A table view underneath lists the same data as rows (chart-plus-table, not chart-only, so the numbers are always readable even with few data points).
2. **Planned vs Executed** — per Schedule's 8 event types: planned hours = sum of scheduled event durations (`start_time`/`end_time`) in range; executed hours = sum of `event_logs` actual duration (`actual_end − actual_start`, skipping `skipped` logs) for events in range. An event with no log yet contributes to "planned" but not "executed" — that gap is the real, meaningful signal this view exists to show, not a bug to paper over.
3. **Productive Hours Heatmap** — 7×24 grid (day-of-week × hour-of-day). Cell value = total `focus_sessions` minutes whose `start_at` falls in that bucket, within range. Color = accent hue at an opacity scaled to that cell's value relative to the max cell in the grid (so the heatmap always uses full contrast regardless of how much data exists).
4. **Subscription-vs-Usage Matrix** — same scatter+table pattern as #1. One dot per `transaction.recurring` row. X = monthly cost (`abs(amount)`). Y = `usage_count`. High-cost/low-usage (upper-left) is the "cut candidate" quadrant, called out with a border/background tint on the quadrant itself, not a scored ranking.
5. **Habit ↔ Output Correlation** — for each active habit: split range days into "habit logged done" vs "not done," compute avg tasks-completed-that-day and avg focus-minutes-that-day for each split, show as a `.dual-bar` pair per habit per metric. No correlation coefficient, no statistics beyond a plain average — matches decision #5.
6. **Calendar Load vs Todo Backlog** — one line-chart with 2 series across each day in range: scheduled hours that day (event durations), and open-task backlog *as of* that day (count of tasks with `status !== 'done'` and `due_date <= that day`, i.e., "due or overdue by then" — the honest computable proxy, since tasks have no `completed_at` history to reconstruct a true point-in-time backlog).
7. **Decay Analysis** — a flat ranked list (not scoped to the date-range filter, since staleness is inherently "how long has NO activity happened," which needs to look further back than the range) of tasks/projects/habits sorted by days-since-last-activity descending: tasks = days since `due_date` if overdue-and-open, else days since `created_at` if never touched; habits = days since last `habit_logs` row; projects = days since the most recent task/event/transaction referencing that `project_id`. Items past a fixed, documented threshold (14 days) get a `.pill.amber` "stale" flag — a real, stated threshold, not a hidden one.
8. **Theme Day Output** — group each day in range by its `theme_days.label` (via day-of-week), then average 3 real metrics per theme group: focus hours, tasks completed, habit consistency (that day's done-count / active-habit-count). Shown as one `.dual-bar`-style row per theme per metric, or a small 3-column table per theme — whichever reads cleaner once real data exists (decided during build, logged in the drift log either way).
9. **Time-Money Rate** — trend line (Habits'-style SVG polyline) of `hourlyRate` (net $ ÷ focus hours), one point per week inside the range, using the *duplicated-and-generalized* version of Finance's formula per decision #6.
10. **Cost per Focus Hour** — same trend-line treatment as #9, paired metric: `fixedSpent ÷ focusHours` per week in range.
11. **Energy Index** — trend line, one point per day in range. Each day's value = unweighted average of 3 real percentages, each capped at 100: (tasks completed that day ÷ tasks due that day), (habits done that day ÷ active habits), (focus minutes that day ÷ 240, a stated 4-hour reference target — not a hidden number). Explicitly documented in the module's own comments as "an average of three real percentages, not a scored or predictive model," per decision #5.

## Scope

1. Sidebar/topbar: add the 5th `custom` segmented button + popover; wire `wireTopbarSegmented()` in `js/app.js` to persist the active range (`Console.db.setPref('analytics_range', ...)`) and restore it on boot.
2. `js/modules/analytics/index.js`: replace the Phase 1 stub entirely. View-tab shell (11 tabs is a lot for the existing `.vtab` row — group into a `.vtab-extra`-style secondary row or a `<select>`-driven view switcher if 11 bare tabs don't fit visually; decide and log during build) plus the 11 view renderers above.
3. New shared CSS: `.dual-bar`, a scatter-plot container + dot, a heatmap grid + cell, reusing `.pane`/`.pane-head`/`.metric`-family/`.pill` everywhere else.
4. No changes to `db.js` — confirm this stays true during build (if any view turns out to need a stored field that doesn't exist, that's a real gap to log, not silently work around).

## Acceptance criteria

- [ ] All 11 views render with real data (or a real, honest empty state — "not enough data yet," matching Finance's Phase 5 "no rate yet" precedent, not mock numbers).
- [ ] Topbar 7d/30d/90d/all/custom is live, persists across reload, and only affects Analytics (verified: switching range on another module's screen, then visiting Analytics, still shows the right range; switching range while on Analytics doesn't alter any other module's own data).
- [ ] Every formula that duplicates a Finance/Schedule calculation produces the same number as the source module would, for an equivalent single-period range (spot-checked during verification, not assumed).
- [ ] No invented CSS class ships without a real rule — this exact gap slipped through code-review twice during Phase 6 (`.metric`/`.mnum`/`.mlbl`, bare `.empty-sub`) before being caught; grep every class name used against `components.css`/`base.css` before calling this phase code-verified.
- [ ] Decay Analysis's 14-day threshold and Energy Index's 240-minute reference are documented inline in the code, not just in this brief.

## Explicitly out of scope

- No predictive/ML anything, anywhere in this phase (decision #5).
- No new IndexedDB store or schema field, even if a "nice to have" surfaces during build — flag it instead (e.g., true point-in-time backlog would need a `completed_at` on tasks; not adding it this phase).
- No retrofitting range-filtering into Today/Tasks/Schedule/Habits/Finance/Focus (decision #2).
- No charting library — everything is hand-rolled SVG/CSS per decision #4.
