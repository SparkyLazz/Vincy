## Scope

Habits = Atomic Habits framework fields per habit, multi-state daily logging, rolling 30-day consistency (no streaks anywhere), quantitative targets, and read-only cross-domain link counts (Focus sessions / Tasks by matching tag ‚Äî the auto-log *write* path is Focus's own job once Focus exists, see exclusions below).

1. **Habit CRUD** ‚Äî `habits` store: `id`, `name`, `identity` (the "I am a ___" statement), `cadence` (`'daily'` or `'n_per_week'`), `cadence_n` (integer, only meaningful when `cadence === 'n_per_week'`, e.g. `3`), `category_label` (free text, e.g. "keystone", "writing", "mindfulness" ‚Äî a label, not a strict enum, same precedent as `theme_days.label` in the Phase 3 brief), `category_color` (one of the existing `.pill` color classes: `teal`/`blue`/`amber`/`plum`/`danger`/`neutral` ‚Äî chosen at creation, not invented per-render), `target` (`{ unit, amount }` or `null` for a simple boolean habit ‚Äî quantitative habits per the features list: pages/minutes/reps/ml/etc.), `cue`/`craving`/`response`/`reward` (Atomic Habits framework strings), `stack_before` (habit-stacking: what precedes this habit), `environment`, `tag` (optional string, for cross-domain matching against `tasks.tags` and a future `focus_sessions` tag), `status` (`'active'`/`'archived'`), `created_at`. "New habit" button + `N` shortcut, and `‚åòE` (edit) on a selected habit, both open the existing `.modal-overlay`/`.modal`/`.modal-field` system built fresh in Phase 3 for Schedule's new-event form ‚Äî reused verbatim here rather than building a second modal shell or cramming ~13 fields into the twopane detail rail. Don't invent new input styling; `.modal-row`/`.input`/`.notes-area` already cover every field shape this form needs.
2. **Daily logging** ‚Äî `habit_logs` store: `id`, `habit_id`, `date` (`YYYY-MM-DD`), `state` (strict enum: `done`/`partial`/`skip`/`miss`), `value` (number or `null` ‚Äî the logged quantitative amount, only used when the habit has a `target`), `note` (optional), `logged_at` (ISO timestamp). One log row per `[habit_id, date]` (upsert on that pair, matches the existing `habit_id_date` index). `D`/`P`/`S` are quick one-key state-setters for today's log on the selected habit (state only, no note/value prompt ‚Äî matches the prototype's `hlog` quick-toggle buttons); `L` opens the fuller log form (state + quantitative value + note) for the selected habit's today entry in the same reused modal shell as #1, same shape as Schedule's own `L` (planned-vs-actual log form). There is no dedicated shortcut for `miss` (the prototype's own `kbd-hints` list only binds D/P/S ‚Äî `miss` stays a UI-only 4th `hlog` button, exactly as the prototype shows it, not an oversight to fix).
3. **Rolling 30-day consistency % ‚Äî no streaks** ‚Äî new shared helper `js/lib/habits.js`: `Console.lib.habits.consistency30(habitId, logs, todayISO, createdAtISO)`. Heuristic (simple, documented here and in the file rather than a scoring model ‚Äî same discipline as Schedule's theme-day-adherence note): walk the window from `max(createdAt date, today ‚àí 29 days)` through yesterday (today itself is excluded until it's actually logged, so an unlogged "not yet" habit today doesn't drag its own number down mid-day ‚Äî matches the prototype's "not yet" `today-val` state). For each day in the window: `done` = 1 point / 1 denom, `partial` = 0.5 point / 1 denom, `miss` or no log at all = 0 point / 1 denom, `skip` = excluded from both point and denom entirely (this is the antifragile mechanic the prototype's own philosophy note describes ‚Äî "missing one day never resets your progress" ‚Äî a skip is a deliberate non-penalized pass, not a miss). `% = round(point / denom * 100)`, or `0` if denom is `0` (a brand-new habit with no eligible days yet). Same helper also exports a day-chain builder (`Console.lib.habits.dayChain(habitId, logs, todayISO, createdAtISO, n)`) used for both the 14-day mini chain (`.hc-chain`) and the 60-day full chain (`.chain-full`), so the two chain renderers can't drift into two different day-state rules.
4. **Cross-domain link counts (read-only this phase)** ‚Äî the prototype's `.cd-links` block (Focus sessions tagged X, Tasks #tag) is real joined data, not mock: count `tasks` where `tags` includes the habit's `tag` (real, Tasks already ships), and count `focus_sessions` where a session's `tag`/`context` matches the habit's `tag` (this will legitimately read as `0` until Focus ships in Phase 6 ‚Äî an honest empty state per the "real empty state, not mock content" rule from Phase 1's own acceptance criteria, not a bug to chase). Writing new `habit_logs` rows automatically when a matching focus session completes is explicitly Focus's job once it exists (see exclusions).
5. **4 views via `.view-tabs-row`**: **Today** (checklist of all active habits' today state ‚Äî reuses the *exact* `.habit-list`/`.habit-row` markup already extracted into `base.css`/`components.css` from Today-dashboard's own habits card in Phase 1, now wired to real per-habit state/value instead of the hardcoded 0%/empty toggle it shipped with), **All habits** (the prototype's shown view ‚Äî `.habits-grid` of `.habit-card`s in the left `.pane`, `.detail-head`/`.detail-section`/`.detail-actions` detail panel in the right `.pane`, exactly as drawn), **Correlations** (no prototype exists for this tab ‚Äî built fresh from existing atomic components per #6 below, same "no prototype existed, built from proven pieces" precedent Phase 3 used for its new event modal), **Archive** (flat `.pane`/`.list-row.habit`-style list of `status: 'archived'` habits with a "reactivate" action ‚Äî also no prototype; `.list-row.habit` already exists in `components.css` with the right 4-column template but had no real consumer before this phase, same "found an unused generic template, first real user" precedent as Phase 3's `.list-row.event`).
6. **Correlations view (new layout, no prototype)** ‚Äî keep it simple, not a scoring model: for each active habit, one `.cd-link`-style row (already styled, reused verbatim) showing its 30-day consistency % next to its linked Focus/Task counts from #4, sorted by consistency descending. This is explicitly the "simple heuristic, logged, not over-built" version ‚Äî richer correlation math (e.g. did high-consistency days correlate with more focus hours) is Analytics' job (Phase 7), not Habits'.
7. **Archive view (new layout, no prototype)** ‚Äî flat list (`.pane` + a `.list-row`-shaped row reusing the existing generic list-row pattern, not a new grid) of archived habits, each with a "Reactivate" button that flips `status` back to `active` (no data loss ‚Äî logs stay intact either way).
8. **Metrics row** (`.metrics-row`, real data, matches prototype's 4 metrics exactly): avg consistency % across active habits (30d), active habit count (`N/N` tracked today = logged-today count over active count), best-performing habit this window, and one "watch" habit (lowest consistency, only flagged if below a simple threshold ‚Äî reuse the prototype's own amber "watch" treatment, don't invent a new warning tier).
9. **Keyboard shortcuts** (scoped to Habits active, palette closed, same collision-avoidance discipline as Phases 2‚Äì3):
   - `N` ‚Äî new habit
   - `J` / `K` ‚Äî navigate selection between habit cards (All habits) or rows (Today/Archive)
   - `L` ‚Äî open the full log form (state + value + note) for the selected habit's today entry
   - `D` / `P` / `S` ‚Äî quick-set today's log state to done / partial / skip for the selected habit
   - `Enter` ‚Äî open the selected habit (moves focus into the detail pane; a no-op beyond that in the All-habits view, since the detail pane already renders on selection, same shape as Tasks' own Enter/click equivalence)
   - `‚åòE` ‚Äî edit the selected habit's framework fields
   - `?` stays inert this phase (Settings' job, Phase 8 ‚Äî same deferral every prior phase logged)
10. **Sidebar Habits count**: already wired in `app.js` (`habitsActive` = count of `status: 'active'` habits) since Phase 1 ‚Äî no change needed, just confirm it still reflects real rows once Habits has real data (same re-check discipline every phase applies to its own sidebar count).
11. **Today dashboard cross-check**: `js/modules/today/index.js`'s `buildHabitsCard` (built in Phase 1 against an always-empty store) currently hardcodes an empty `.htoggle` and `width:0%` fill ‚Äî re-verify and fix it to show each active habit's real today state (`.htoggle.done`/`.partial`/`.skip`/`.miss`) and real rolling 30-day % (via the new `Console.lib.habits.consistency30` helper, not a second copy of the math), same re-check discipline Phase 2 applied to Quick Capture and Phase 3 applied to the calendar card.
