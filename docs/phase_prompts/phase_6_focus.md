# Console — Phase 6: Focus Sessions

> Ships against the fidelity bundle: `css/tokens.css`, `css/base.css`, `css/components.css`, `docs/visual_contract.md`, this brief. **No prototype exists for Focus** (`Console_Features_List.md` §6 says "not yet prototyped") — every view is built fresh from already-proven atomic components, same precedent Phase 3 used for its new event modal and Phase 5 used for Transactions/Envelopes/Subscriptions/Reports. Follow `docs/Console_Workflow.md` — no rushing, fully verified (code + browser, both themes) before Phase 7 starts.

## Design decisions — read before writing any layout/component code

Since there's no prototype to extract from, these are genuine design calls, not extraction conflicts. Logged here so the reasoning survives even if the code is read in isolation.

1. **Not a "flush" module — reuses Finance's plain `.content`/`.content-inner` layout, not Tasks/Schedule/Habits' twopane shell.** None of Focus's 3 views are a natural list+detail split: Timer is a single centered control panel, Log is a flat chronological list, Summary is a stat block. Same call Phase 5 made for the same reason. `Console.modules.focus` does **not** set `layout: 'flush'`.
2. **Session-type tag reuses Schedule's exact 8 `TYPES`** (`deep_work/meeting/admin/exercise/break/social/errand/sleep`), not a bespoke Focus-only list. A few of those (`meeting`, `sleep`) will rarely get used as a *timed focus session* type, but reusing the identical locked taxonomy means a future Analytics view (Phase 7, "Planned vs Executed" / "Theme Day Output") can directly compare scheduled-vs-logged hours per type with no mapping table — more valuable than a prettier bespoke list. Same "don't invent a second taxonomy when a locked one exists" discipline the project already follows for event types.
3. **`focus_sessions` fields, finalized now** (the store's `keyPath`/indexes were already locked in `db.js` since Phase 1 — `start_at`, `type`, `task_id`, `habit_id` — but the full row shape was never written down until a real module needed it): `id`, `start_at` (ISO datetime), `end_at` (ISO datetime | `null` while active — already the contract every other module's read-only join assumed, see `js/app.js`'s `refreshFloatingTimer`), `mode` (`'pomodoro'|'freeform'|'stopwatch'`), `type` (one of Schedule's 8 types, #2 above), `task_id`/`habit_id`/`event_id` (trigger refs, independently nullable — standalone = all three `null`), `trigger_label` (denormalized snapshot of the linked task/event/habit's title *at session start* — see #4), `target_min` (pomodoro defaults to 25, editable; stopwatch-with-target is user-set; freeform is `null`), `paused_at` (ISO | `null`), `paused_ms` (accumulated pause duration), `strict` (bool, soft-strict mode for this session), `duration_min` (computed once at End and stored — the one exception to "always compute, never store," justified because a completed session's duration is a historical fact, not a live-changing derived value, unlike Finance's `envelopeSpent` or Habits' `consistency30`), `notes`.
4. **`trigger_label` is a denormalized snapshot, not a live join — a deliberate simplification.** The floating timer (`js/app.js`) renders from any screen without loading Focus's module-internal cache, so it needs the trigger's display name without a 3-way join across `tasks`/`events`/`habits` on every tick. Snapshotting the label at session start means it won't retroactively update if you rename the linked task mid-session — an acceptable, logged tradeoff for a personal app, not a silent gap.
5. **Soft strict mode = a `body` class, not a route guard.** "Dims other UI, navigation still allowed" (per `Console_Features_List.md` §6) means sidebar/topbar get reduced opacity via a new `.focus-strict` class toggled on `<body>`, restored to full opacity on `:hover` so the user can still see what they're clicking — not `pointer-events: none` anywhere, since navigation must stay possible. The class is set/cleared by the session start/end actions (in `js/app.js`, alongside the floating timer, not scoped to the Focus module's own lifecycle) so it persists correctly if the user navigates away from Focus while a strict session is still running, and is re-applied on boot (`refreshFloatingTimer`) if an active strict session is found in the DB.
6. **Trigger picker lives inside Focus's own Timer view — no "Start focus" buttons added to Tasks/Schedule/Habits.** Modifying three already-verified modules to add cross-launch buttons is real scope creep for one phase. The one exception: Today dashboard's `buildSuggestedFocusCard` (stubbed since Phase 1, cross-checked now per the same "re-check Today's card once the real module exists" discipline Phases 2–5 all followed) gets a real "Start focus" link that jumps to `#/focus/task/<id>` — the router already ignores hash sub-segments beyond the first (`router.js`'s `parseHash` does `raw.split('/')[0]`), so this needs zero router changes; Focus's own `init()` reads `location.hash` directly to detect and pre-fill a pending trigger.
7. **`.list-row.focus`, a new column template** (no unused pre-existing template to reuse here, unlike Phase 3/4/5's `.list-row.event`/`.habit`/`.report` discoveries — Focus is genuinely new). 4 columns: time-range / title+meta (mode pill + type dot) / trigger pill / duration (`.num-mono`, right-aligned) — same visual grammar as `.list-row.event`, adapted.

## Scope

Focus Sessions = a timer (Pomodoro / Freeform / Stopwatch-with-target) that can be triggered standalone or from a task/event/habit, a floating always-visible indicator, and a real session log/summary — no scoring or correlation math (that's Analytics', Phase 7).

1. **Timer view (default)** — mode picker (`.pill-segment`, 3 options), trigger picker (segmented "Task / Event / Habit / Standalone", then a `<select>` populated from that store's open/upcoming real rows), session-type `<select>` (Schedule's 8 types, reused per Design Decision #2), soft-strict checkbox, target-minutes field (only shown for `mode: 'stopwatch'`; `mode: 'pomodoro'` shows a fixed-but-editable 25 default; `mode: 'freeform'` shows nothing). Start creates a `focus_sessions` row (`end_at: null`) and begins a live `mm:ss` (or `hh:mm:ss` past 60 min) display, ticking every second via `setInterval`. Pause/Resume toggle `paused_at`/accumulate `paused_ms` (elapsed display freezes while paused). End writes `end_at`, computes and stores `duration_min = (end_at − start_at − paused_ms) / 60000`, clears the active state. Below the control panel: a compact "today's sessions" mini-list for immediate context (reuses the Log view's row renderer).
2. **Log view** — full session history via `.list-row.focus` (Design Decision #7), grouped by date like Schedule's Agenda (`agenda-date-label` reused). Each row: time range, mode pill, type dot + label, trigger pill (task/event/habit title, or "standalone"), duration. Click opens an edit modal (type/trigger/notes only — start/end/duration are historical facts, not editable, matching the "don't let the log rewrite what actually happened" spirit of Schedule's planned-vs-actual logging) or delete.
3. **Summary view** — real-data-only, no scoring model (Analytics' job per exclusions below): today + this-week total focus time, a breakdown by `type` (reusing Schedule's 8 types, sum of `duration_min` per type — real numbers, same "simple real trend, no model" precedent as Finance's Reports view), and a breakdown by trigger-kind (task/event/habit/standalone session counts this week).
4. **Floating active-session indicator (`js/app.js`, not scoped to the Focus module)** — the container already exists from Phase 1 (`#floating-timer`, currently static). Wired to real state: `ftmode` = session mode, `fttitle` = `trigger_label` or the type's label if standalone, `fttime` = live `mm:ss` ticking independently of whether Focus's own module is currently mounted (a `setInterval` owned by `app.js`, reading the active session fresh from `focus_sessions` — same "works from any screen" requirement as the soft-strict body class). Pause/End buttons become real and functional from any screen, not just while viewing Focus.
5. **Cross-domain auto-linking** — a session with `habit_id` set counts toward that habit's `focusLinkCount` (already implemented and counting real numbers since Phase 4 — this phase is what finally makes that number non-zero, not new code); a session with `task_id` set doesn't change the task's own state (marking it done is still a manual Tasks action — auto-completing a task because a timer ran is out of scope, see exclusions).
6. **Today dashboard cross-check — `buildSuggestedFocusCard`.** Stubbed since Phase 1 with static "Activates once Tasks and Schedule (Phases 2–3) have data" copy — both have had real data since Phase 2/3, so this was stale the whole time, same class of gap Phase 2 found in Quick Capture's stale copy. Real (simple, not scored) heuristic: surface the single highest-priority open task without a logged focus session today, with a "Start focus" link to `#/focus/task/<id>` (Design Decision #6). If no open tasks exist, render the existing honest empty state instead of a fake suggestion.
7. **Keyboard shortcuts** (scoped to Focus active, palette closed, same collision-avoidance discipline as Phases 2–5):
   - `S` — start (only when idle)
   - `P` — pause/resume (only when a session is active)
   - `X` — end session (only when active — avoids colliding with Tasks' own `X`, which isn't active at the same time since only one module is mounted)
   - `J`/`K` — navigate the Log view's row selection
   - `Enter` — open the selected row's edit modal
   - `⌘E` — edit selected (same key as every prior phase's own binding)
8. **Sidebar Focus nav count**: `app.js`'s `refreshNavCounts` currently has no Focus line (the nav item itself has no `.nav-count` span at all, unlike Today/Tasks/Schedule/Habits/Finance). Add one — real count of focus sessions logged today (mirrors Today's own "todayOpen" semantics), plus the `.nav-count` span in `index.html`.

## Data model touched

- `focus_sessions` (real reads/writes for the first time): full row shape per Design Decision #3. `keyPath`/indexes already correct in `db.js` (`start_at`, `type`, `task_id`, `habit_id`) — no schema change needed. `event_id` is a new non-indexed field (same precedent as `categories.group`/`habits.tag` being undocumented-until-their-phase fields) — not indexed since nothing needs to query focus sessions by event yet.
- `tasks`, `events`, `habits` (read-only joins, no writes): for the trigger picker's real options list and `trigger_label` snapshotting.

No schema/index changes to `db.js`.

## Acceptance criteria

- [ ] Start/Pause/Resume/End all write real `focus_sessions` rows; `duration_min` is computed correctly accounting for accumulated pause time.
- [ ] All 3 timer modes work: Pomodoro shows/respects a target (default 25 min, editable), Freeform has no target, Stopwatch-with-target shows progress against a user-set target without auto-stopping at it.
- [ ] Trigger picker offers real Task/Event/Habit rows (not mock data) plus a Standalone option; the chosen trigger's id and a `trigger_label` snapshot are stored on the session.
- [ ] Soft-strict mode visibly dims sidebar/topbar via a `body` class, restores full opacity on hover, and never blocks navigation clicks.
- [ ] Floating timer shows real live-ticking state from any screen (not just while Focus is mounted), and its Pause/End buttons work from any screen.
- [ ] Log view groups real sessions by date, `.list-row.focus` renders all 4 columns correctly, edit modal only allows changing type/trigger/notes (not start/end/duration).
- [ ] Summary view shows real today/this-week totals and real per-type / per-trigger-kind breakdowns — no fabricated or scored numbers.
- [ ] Habits' `focusLinkCount` (already-existing code since Phase 4) now returns real non-zero counts once sessions with `habit_id` exist.
- [ ] Today dashboard's Suggested Focus Blocks card shows a real top task with a working `#/focus/task/<id>` deep link, or the honest empty state if there are no open tasks.
- [ ] `S`/`P`/`X`/`J`/`K`/`⌘E` all work while Focus is active, none fire while `⌘K` is open, none collide with another built module's own bindings.
- [ ] Sidebar's Focus nav count reflects real data; `index.html` gets the missing `.nav-count` span.
- [ ] Full `docs/visual_contract.md` checklist passes for every Focus view, both themes.
- [ ] No inline styles beyond genuinely dynamic per-instance values (the live timer's own text content is not a style, so this phase should have close to zero legitimate exceptions — flag anything that shows up).

## Explicitly out of scope for this phase

- **Auto-completing a task/habit because a focus session ran** — cross-domain *counting* (Habits' `focusLinkCount`) is in scope; cross-domain *state mutation* (marking a task done, auto-logging a habit as `done`) is not — those remain manual actions in their own modules.
- **"Start focus" launch buttons embedded in Tasks/Schedule/Habits' own UI** — Design Decision #6's one exception (Today's suggested-focus card) is the only cross-module UI surface this phase adds.
- **Scored productivity/energy detectors** (`productive_hour_misalignment`, Energy Index) — Insights'/Analytics' (Phase 7) job; this phase's Summary view is real-but-unscored, same split Phase 4 drew for Correlations vs. future Habit↔Output.
- **Notification/sound alerts on Pomodoro target reached** — a real desktop notification API integration is a nice-to-have, not required; a visual highlight when `target_min` is reached is sufficient.
- **Multiple concurrent sessions** — only one active (`end_at: null`) session at a time; starting a new one while one is active is blocked (End or discard the current one first), matching the floating timer's own singular-indicator design.
